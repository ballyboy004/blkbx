## 2026-03-19 BLACKBOX as Environment — Motion System Direction
- Decided BLACKBOX should feel like a spatial environment, not a dashboard
- Every interactive surface should respond to presence: hover depth, cursor-reactive tilt, ambient idle motion
- GSAP and Framer Motion are both in the stack and should be used to their full potential — current usage is conservative relative to what's possible
- Planned: dedicated motion/environment pass after launch blockers are cleared
- Specific directions to explore: mousemove-driven card tilt (rotateX/Y with perspective on mission card), z-depth shadow lift on hover surfaces, opacity wave stagger across text lines, fog-lift where non-focused elements recede when something is active, clip-path light sweep across bold text
- Current implementation: z-depth layered textShadow lift on mission card task title
- This is a platform-wide direction — every card, chip, and interactive element should eventually participate in the environment feel

## 2026-03-15 — Campaign Module Architecture

### DECISION: State-driven intelligence principle locked
- Database + module logic = source of truth. AI assists generation only.
- All AI outputs must be persisted as structured data (content_pieces, campaign_tasks).
- The AI console is secondary — it explains, it does not control campaign progression.
- Campaign progression is deterministic: resolveCampaignState, buildMissionCardData, resolveWorkspaceChips all operate on DB state with no AI calls.

### DECISION: Announcement asset type replaced with curator pitch
- "Announcement" was abstract and had no real destination channel.
- Curator pitch is universal, high-leverage, and hard to write well without context.
- Revised asset set: curator_pitch, captions, email, press_release.
- Asset generation is archetype-aware via resolveAssetStrategy(profile).

### DECISION: Campaign module UI direction locked — dark Swiss precision
- Rejected visionOS glass aesthetic (requires vivid background color to work; fails on dark flat backgrounds).
- Adopted: flat near-black background (#0c0c0e), elevated dark cards (#161618 / #111113), IBM Plex Mono for labels, Inter Bold for directive text.
- NEXT MOVE directive uses Inter Bold at 28-32px — the dominant element on the page.
- No backdrop-filter, no gradients, no glow. Precision tool aesthetic.

### DECISION: Workspace layout — Mission / Work / AI three-layer structure
- Mission card: campaign state, current phase, next move directive, progress.
- Work card: assets waiting review (status = 'generated'), links to review route.
- AI console: fixed bottom, minimized by default, auto-expands on chip click.
- This order is non-negotiable — AI is tertiary, never primary.

### DECISION: Asset strategy is visibility-style driven
- resolveAssetStrategy(profile) determines which assets to generate before any generator runs.
- anonymous/music-first → no assets generated.
- scarce/mysterious → curator pitch only.
- event-driven → pitch + captions.
- community-driven → captions + email.
- consistent/present → full set.

### DECISION: Release task set grounded in artist career course material
- Tasks now map to real release cycle phases: pre-release, launch week, post-release.
- Pre-release: Spotify for Artists editorial submit, curator outreach, pre-save, content teasers.
- Launch: all-platform simultaneous drop, fan email first, engagement push, saves > streams.
- Post: day 2-7 content, curator follow-ups, milestone posts.
- Source: Artist Playbook modules 2-5 ingested as campaign execution framework.

## 2026-03-15 — Onboarding Rebuild

### DECISION: Onboarding rebuilt as adaptive question tree
- Replaced 6 static panels with dynamic panel sequence driven by buildPanelSequence(role, projectStatus).
- Panel set is determined by user answers — producers see a different path than artists.
- Artist with ready release: role → sound → project_status → readiness → goals → constraints → interpretation.
- Producer path: role → sound → producer_goals → catalog → blocker → constraints → interpretation.

### DECISION: Onboarding ends with AI interpretation panel
- Final panel shows a 2-3 sentence BLACKBOX read of the user's situation.
- Generated by Claude Haiku via /api/onboarding/interpretation.
- Lowercase, grounded, specific to their answers. No hype, no generic advice.
- Ends with one sentence about what BLACKBOX will focus on first.

### DECISION: Smart redirect after onboarding
- Artist with project_status = 'ready' or 'in_progress' → createCampaignFromProfile() → redirect to /campaign/[id].
- All other paths → redirect to /dashboard.
- Campaign title seeded from genre_sound or context field.
- Rationale: artist with a release ready should land in a workspace immediately, not an empty dashboard.

### DECISION: Campaign intake lives in onboarding, not a separate form
- Onboarding = who you are + what you're doing right now.
- Campaign form still exists for subsequent campaigns.
- New profile fields: role, project_status, readiness_checklist (jsonb), campaign_goals (jsonb), primary_blocker.

### DECISION: BLACKBOX must be role-flexible from the start
- Platform is not artist-only. Producers, songwriters, both/hybrid users are first-class.
- Asset sets, task generation, and AI context all adapt to role.
- "Campaign type" dropdown rejected — BLACKBOX infers campaign type from answers, not forced selection.
